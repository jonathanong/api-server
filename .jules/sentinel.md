## 2026-06-01 - [Malformed URL 500 Error Log Pollution]
**Vulnerability:** Node's `new URL(url)` throws a synchronous error for malformed URLs (e.g. `http://[/foo`). If `req.url` contains an absolute URL and is parsed without a try-catch, it throws an uncaught exception during request handling, resulting in an unhandled 500 Internal Server Error rather than a 400 Bad Request.
**Learning:** `req.url` can be arbitrary user input, and when evaluating it as an absolute URL, standard library parsers (like `URL`) are strict and throw. Unhandled throws early in request routing logic cause 500s, polluting error logs and creating a vector for DoS via noise/alert fatigue.
**Prevention:** Always wrap `new URL()` in a `try-catch` when operating on `req.url`, and re-throw with `{ status: 400 }` to ensure malformed requests are properly rejected without causing server-side exceptions.

## 2025-02-20 - [Denial of Service via Unbounded Body Streams]
**Vulnerability:** The server enforces a body size limit (default 1MB) for incoming requests. However, when the limit is exceeded, it only rejects the promise and drains the rest of the stream to keep the HTTP connection alive. A malicious client could send an infinite stream of data (e.g. chunked encoding without ending), forcing the server to keep reading and discarding data indefinitely, leading to resource exhaustion (CPU/bandwidth) for a single connection.
**Learning:** Draining streams after a 413 Payload Too Large might keep connections reusable for well-behaved clients, but it allows malicious clients to keep a connection open and sending data forever, tying up server resources. Connection keep-alive is not worth the DoS risk for oversized payloads. Furthermore, checking Content-Length allows early rejection without reading any body data.
**Prevention:** Always forcefully close the connection (`Connection: close` and `req.pause()`) when rejecting a request for exceeding the payload limit. Fast-fail if the `Content-Length` header exceeds the limit before consuming the stream.
